Design: crash-stem impact score

2026-07-16 · bugs 265/266 · supersedes the 2026-07-15 recalibration of src/analyzer/crash_accents.py

Goal

Make audible "hero" cymbal crashes actually reach generated sequences: fix the stale-cache silent skip and the peak-suppression bug, and replace the overfit full-mix treble detector with a cymbal-stem isolation score validated 6/6 on user-confirmed Dream On ground truth.

Evidence base (2026-07-16 prototyping)

Signal sourceResult on 6 confirmed Dream On crashes
Full mix, ≥6kHz band (shipped approach's signal)50.85s not even a candidate; 190.2s rank 26. Unusable.
Full drum-kit stem (what demucs htdemucs_6s provides)Full band: misses 51.1s, no gap. ≥6kHz: one miss, one rank-19, false positives interleaved.
Combined cymbals stem (crash+ride+hh summed)5/6 in top 5, gap 7.44 → 5.55; 103.7s falls to rank 9.
Crash-isolated stem6/6 as the top 6, clean gap 10.26 → 6.19.

Scoring used throughout: per envelope peak, score = log1p(isolation) × log1p(wash_area / median_wash_area), where isolation = peak height over the median stem envelope in the preceding 8s (a hero crash erupts out of local cymbal-silence; the outro's wall-of-cymbals moments — which swamped every full-mix formulation — score near zero), and wash_area = area above the pre-crash background, bounded 6s (a hero crash stays bright; ordinary hits are narrow). Raw envelope area alone was tested and rejected: on this song it degenerates into "how loud is the outro".

Approach

  1. Cymbal separation stage — new src/analyzer/drum_stems.py: separate_cymbals(drums_audio, sr, cache_dir) -> np.ndarray | None. Round-1 implementation: a drumsep-class drum-kit separation checkpoint loaded through the existing demucs/torch optional dependency (one model download, cached like other torch hub assets). Output cached as .stems/<md5>/drums_cymbals.wav beside the existing demucs stems. Returns None (never raises) when demucs/torch or the checkpoint is unavailable.
    Alternative considered: the audio-separator pip package with a roformer drum-kit checkpoint (closer to the mvsep MT-7 quality used in prototyping). Rejected for round 1: a new heavyweight dependency before the demucs-checkpoint route's quality is proven insufficient. Task 2 explicitly validates local cymbal-stem quality against the user's mvsep reference stems; if ranking degrades materially, revisit.
  2. Detector rewrite — detect_crash_accents(cymbals, sample_rate, full_mix) -> list[TimingMark] (signature change; sole production caller is orchestrator.py). Pipeline: full-band STFT envelope of the cymbals stem, 3-frame smoothing → find_peaks with a short 3s distance (candidate spacing only) → per-candidate isolation + wash-area score as above → gates:
    • absolute score floor — final: 7.0, calibrated on a 6-song local panel (drumsep stems, 2026-07-16): Dream On's confirmed crashes score 7.51–10.61 vs 6.17 for its first non-confirmed candidate; panel yields 6/3/2/1/0/0 marks (Dream On / 1999 / Chattahoochee / Raining Tacos / Smooth Criminal / 8bitSlash). Smooth Criminal's quiet-intro transient (score 13.3) is correctly killed by the pre-RMS guard;
    • full-mix pre-transient RMS floor ≥ 0.4 of median (kept verbatim from the shipped fix — the cold-open false positive must stay dead);
    • 10s min-gap enforced post-scoring (keep the higher-scored of any conflicting pair) — this is the bug-266 fix; the constraint no longer suppresses candidates before they are scored;
    • hard cap 6 marks/song (raised from 5 per user decision 2026-07-16 — Dream On legitimately has 6 confirmed crashes; "a handful per song is the ceiling" still holds).
    No cymbals stem → return []. Alternative considered: keep the full-mix treble detector as a fallback when stems are unavailable — rejected: it demonstrably produces 5 wrong marks on its own validation song; for a rare-by-design feature, zero marks beats wrong marks.
  3. Cache invalidation — bump SCHEMA_VERSION to "2.1.0". Pre-feature caches (missing field) and 07-14..07-16 caches (broken detector's marks) both re-analyze. This requires fixing six brittle exact-string readers (listed below) to a major-version check (schema_version.startswith("2.") semantics via src/schema_check.py), which also un-mines the field for every future minor bump. Alternative considered: targeted invalidation (if "crash_accents" not in data: return None) without a version bump — rejected: it would not invalidate the 07-14..07-16 caches that contain the broken detector's five bogus marks, and it leaves the exact-string landmines in place.
  4. Review-UI visibility — emit the marks additionally as a derived L0 TimingTrack (name crash_accents, algorithm derived), same pattern as eighth_notes, so the track is visible in the review UI and exports. HierarchyResult.crash_accents stays the generator's source of truth (placement code untouched).
