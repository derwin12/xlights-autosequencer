OpenSpec design · paste-lyrics-fallback

Design: paste-lyrics fallback

Let a user paste lyrics text by hand, as an optional fallback for songs no provider in the syncedlyrics allowlist has indexed — confirmed for a brand-new release ("Shake the Snowglobe", Gwen Stefani) where every provider returned nothing while a well-known song succeeded through the same code path. The pasted text flows into the exact same lyrics_text_override cache path an automatic provider match already uses, so no downstream consumer needs to change.

Approach

Three additive pieces, no existing behavior changed:

  1. parse_pasted_lyrics(text) in src/analyzer/synced_lyrics.py — validates/previews user-supplied text using the same parse_lrc + plain-line + _is_credit_line path check_synced_lyrics_with_text already runs a provider result through. Returns {found, reason, line_count, preview, source: "pasted"}. Empty/whitespace-only text returns {found: false, reason: "empty"}.
  2. POST /api/v1/lyrics/paste in src/review/api/v1/analysis.py — body {title, artist, lyrics_text}. Writes the raw text into the existing _lyrics_cache[(title, artist)] dict (the same one check_lyrics populates and start_analyze reads), keyed by the exact title/artist strings sent — the same convention check_lyrics already uses, so both routes agree on cache keys without new coordination logic.
  3. PasteLyricsDialog — a new modal component (textarea + Save/Cancel), following the ReanalysisDialog overlay/dialog CSS-module convention already in the codebase. A "Paste Lyrics" button sits next to the existing "Check Lyrics" button on the Analyze screen's pre-analysis metadata panel. Submitting the dialog calls the new endpoint and feeds the response into the same lyricsCheckResult state handleCheckLyrics already populates, so the ✓/✗ display is shared code, not a parallel implementation.

Nothing in build_song_story, get_boundary_refinement_inputs, or section classification/merging changes — pasted text is indistinguishable, at the consumer layer, from a provider's untimed plain-text result (the existing fallback path when parse_lrc finds no LRC tags already exists and already feeds find_chorus_body).

Alternative considered

Extend POST /api/v1/lyrics/check with an optional lyrics_text field: when present, skip the provider search entirely and just validate/cache the given text.

Rejected. It would fold two different behaviors (search-the-internet vs. accept-what-the-user-typed) into one route behind a conditional, which muddies what "check lyrics" means for logging and for the three existing tests (tests/review/test_api_analysis.py:260-306) that assert the route always calls check_synced_lyrics_with_text. A dedicated /lyrics/paste route keeps each endpoint doing one job, matching how metadata (PATCH), analyze (POST), and lyrics/check (POST) are already separate single-purpose routes in this file.

Regression surface

  • _lyrics_cache (analysis.py:36) — currently written only by check_lyrics. The paste route becomes a second writer. Sole reader: start_analyze's lookup by (lib_title, lib_artist) (analysis.py:422-428) is provenance-agnostic (just reads whatever text is cached for that key) — no change needed there.
  • check_synced_lyrics_with_text, check_synced_lyrics_available, get_boundary_refinement_inputs — signatures unchanged, no new callers added.
  • No JSON/XML schema field changes. /lyrics/paste's response reuses /lyrics/check's exact {found, reason, line_count, preview} shape plus one additive source field; existing frontend code that reads this shape (only handleCheckLyrics's state setter) doesn't need to change to keep working, though the UI will use source to label the result "Pasted" vs found-by-search.

Historical echoes

  • 2026-07-11, Genius removal (docs/segment-classification-changelog.md): this project does not access genius.com in any form. Pasting preserves that boundary — the user's browser fetches whatever page they choose; the app never does.
  • bug-414 (.wolf/buglog.json, 2026-07-20): a provider (NetEase) injected credit lines (作词 : ...) as ordinary timed lines, contaminating the lyrics timeline before _is_credit_line was added. Copy-pasted text from a lyrics site is at least as likely to carry an attribution block. Routing pasted text through the existing parse_lrc/plain-line path (both already filtered by _is_credit_line) avoids reintroducing this rather than writing a second, unfiltered parser.
  • bug-406 (.wolf/buglog.json, 2026-07-19): lyrics lookups silently failed against filename-derived title + "Unknown" artist because search fired before the user could correct metadata; the fix added the title/artist review step ahead of "Check Lyrics". "Paste Lyrics" lives in that same reviewed panel and keys into _lyrics_cache by the same confirmed (title, artist) pair — no new metadata-timing issue.

Out of scope

  • The syncedlyrics provider allowlist or search logic — unchanged.
  • Section detection, merging, or role classification — unchanged; per docs/segment-classification-changelog.md's mandatory-changelog rule, no entry is needed there since this changes only where input text originates, not how it's used.
  • Fixing the Musixmatch 401 / Megalobiz timeout issues noted separately during investigation of this bug report — unrelated provider-health issues, not blocking this fallback.
