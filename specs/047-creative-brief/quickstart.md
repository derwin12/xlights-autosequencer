# Quickstart: Creative Brief Manual Verification

**Feature**: 047-creative-brief | **Date**: 2026-04-14

This walkthrough exercises the end-to-end Brief tab flow on a real song. Run it
once after implementation to validate that every FR and SC in the spec holds on
a developer laptop.

## Prerequisites

- Review server running locally: `xlight-analyze review` (serves the workspace
  shell on port 5173 by default).
- At least one song in the library with a completed `_analysis.json` that
  detects ≥ 8 sections (a 3-minute pop song fixture works).
- Layout configured in the grouper (otherwise Generate is disabled — expected).
- Brief JSON does **not** yet exist for this song (delete
  `<audio>_brief.json` if it's there from a prior run).

## 1. Cold open — defaults populate from library

1. Open `http://localhost:5173/song/<source_hash>`.
2. Click the **Brief** tab.

**Expect**:
- Every axis control is visible on one scrollable page.
- Every axis selector shows "Auto" (genre/occasion may show ID3-derived values).
- Every axis has a one-line hint directly below it, not hidden behind a tooltip
  (FR-051, US6).
- Per-section overrides table shows one row per detected section, each defaulting
  to "Auto" (FR-020, US5 AC-1).
- Generate button at the bottom is enabled.
- Tab badge shows no "Unsaved" indicator.

## 2. Verify Mood smart-defaults (client-side Phase 3 behavior)

1. Set every sibling control back to "Auto" (they already are on cold open).
2. Change **Mood intent** from "Auto" to **"Dramatic"**.

**Expect** (per research.md §2):
- Transitions flips to "Dramatic"
- Accent intensity flips to "Strong"
- Variation flips to "Focused"
- Palette flips to "Restrained"
- Duration flips to "Flowing"
- Each affected control shows a small "via Mood" chip.

3. Change **Variation** explicitly to "Varied".
4. Change **Mood** to "Party".

**Expect**:
- Transitions, Accent, Palette, Duration re-adjust to the Party row of the table.
- **Variation stays on "Varied"** — the explicit user pick is never overridden.

5. Change **Mood** back to "Auto".

**Expect**:
- All siblings previously on "via Mood" revert to "Auto".
- Variation stays on "Varied" (still explicitly set).

## 3. Submit and persist

1. Pick **Mood = Dramatic**, **Variation = Varied**, **Palette = Restrained**.
2. In the per-section overrides table, set section 2 to a specific theme slug
   (e.g. "witching-hour") and section 5 to another (e.g. "arctic-sunrise").
3. Click **Generate**.

**Expect** (per FR-040, US4):
- Browser network tab shows:
  - `PUT /brief/<source_hash>` with the full Brief JSON (200 OK).
  - `POST /generate/<source_hash>` with a sparse body — `transitions: "dramatic"`,
    `focused_vocabulary: false`, `embrace_repetition: false`,
    `palette_restraint: true`, `theme_overrides: {"2": "witching-hour", "5": "arctic-sunrise"}`,
    and `mood_intent: "dramatic"`. Returns 202 with `{job_id, status: "pending"}`.
- Workspace auto-switches to the **Generate** tab.
- Job progress appears; generation completes within a minute.
- On disk: `<audio>_brief.json` exists, contains schema version 1 and the chosen
  values; `updated_at` is a fresh timestamp.

## 4. Verify the sequence reflects the Brief

1. Download the generated `.xsq` from the Generate tab.
2. Inspect the plan JSON the generator emitted (or re-run with `--dump-plan`).

**Expect** (SC-003):
- `config.palette_restraint == True`
- `config.focused_vocabulary == False`
- `config.transition_mode == "dramatic"`
- `config.theme_overrides == {2: "witching-hour", 5: "arctic-sunrise"}`
- Section 2's `SectionAssignment.theme.slug == "witching-hour"`.
- Section 5's `SectionAssignment.theme.slug == "arctic-sunrise"`.
- Other sections use auto-assigned themes.

## 5. Persistence across server restart

1. Stop the review server. Restart it. Re-open the song's workspace.
2. Click **Brief**.

**Expect** (FR-031, US3, SC-004):
- Every control shows the values from step 3, not "Auto".
- Per-section overrides for rows 2 and 5 show the selected theme slugs.
- Last submitted timestamp shown at top of form.
- First paint with all controls populated completes visibly within ~300ms
  (SC-009 — eyeball check, no stopwatch needed).

## 6. All-Auto submit matches legacy behavior

1. Click **"Reset to Auto"**. Every control flips to "Auto". Click **Generate**.

**Expect** (SC-002):
- Network tab shows POST body is either empty or contains only `mood_intent: "auto"`.
- The resulting `.xsq` is byte-identical (modulo timestamps) to a sequence
  generated by the dashboard Generate flow on the same song (spot-check by
  comparing the plan JSON's effective config: every creative field matches the
  library default).
- Brief JSON on disk is rewritten with every field as `"auto"` and
  `per_section_overrides: []`.

## 7. Validation errors surface inline

1. Open browser devtools, simulate a malformed POST (e.g. via `fetch`) with
   `{"transitions": "explosive"}`.

**Expect** (FR-042):
- Server returns 400 with `{field: "transitions", error: "..."}`.
- Brief tab renders the error inline beside the Transitions control, not as a
  top banner.
- No job is created in `_jobs`.

## 8. Story-review no longer influences generation

1. Open `/story-review` for the same song. Change the theme preference on
   section 0 to something different from the Brief's override. Save.
2. Re-open the song workspace → Brief. **Do not** change anything. Click
   Generate.

**Expect** (clarification round 2, FR-041):
- Generated sequence uses the **Brief's** overrides and genre/occasion/transition,
  not the story-review file's.
- `_story_reviewed.json` is still on disk and unchanged by this flow — the
  story-review page still owns its own file. The Brief just doesn't read it.

## 9. Stale theme slug handling

1. Delete the custom theme `witching-hour` from `~/.xlight/custom_themes/`.
2. Re-open the Brief tab.

**Expect** (FR-033 edge case, US3 AC-4):
- Section 2 row shows "Auto" with a warning chip ("Theme no longer exists").
- Section 5 row is unaffected.
- Submitting now clears the stale override from the Brief JSON.

---

If any step above fails, the feature is not done. Every listed expectation
traces back to a numbered FR / User Story / Success Criterion in `spec.md`.
